# Requirements Grooming

Analyzes the requirements from the context file through a QA grooming
lens, finds problems, aligns with the user, and produces the final
numbered list of requirements — input for the next skills.

## Inputs

The input is the context file `<ISSUEKEY>-context.md` created by the
context-collection skill (task-context). In the same chat the file is
available in the working directory automatically. If the chat is new —
the user uploads the file into the chat.

The context-collection skill in this chain is **task-context**.

Data sources for this skill:
- the context file (source of truth), including ALL of its requirement
  sections: "Requirements", "Additional requirements (from comments)",
  and any "⚠️ Conflicts to resolve" section
- the attachments the context file references, if they are needed to
  understand the requirements
- the user's answers to grooming questions

The context file may contain warning notes from the context-collection
skill (for example "⚠️ No Confluence acceptance-criteria page linked").
Read them: a missing acceptance-criteria source means coverage is weaker
than ideal, so groom more aggressively for uncovered requirements and
carry the warning into the final file's Goal/notes.

Do not go to the tracker, do not use external tools, do not search the
internet. The context file is the only source of truth.

The context file is read-only. Do not change, rewrite, or clean its
content.

If the context file is corrupted, duplicated, or internally
contradictory — stop and tell the user the context needs to be
regenerated by the context-collection skill.

## Rules

- All communication and all output-file content — in English.
- Keep chat messages short.
- Analyze only what is in the context file. Do not infer, do not
  supplement, do not interpret requirements on your own.
- Do not turn ambiguity into fact.
- Preserve the original meaning of the requirements during analysis. Do
  not improve, do not rephrase, do not simplify the wording on your
  own. A requirement may be rewritten only after alignment with the
  user.
- Related context from the tracker is background only. Do not create
  findings just because a related task has different wording or
  additional requirements.
- Do not generate findings artificially. If a requirement is clear and
  covered — it is covered.
- After saving the file — stop. Do not continue into a checklist, test
  cases, or planning.

## Numbering requirements

Assign a sequential ID — REQ-1, REQ-2, REQ-3... — to each item in BOTH
the "Requirements" section and the "Additional requirements (from
comments)" section of the context file. Comment-derived requirements are
real requirements: number and groom them exactly like the rest. Number
the "Requirements" items first, then continue the sequence through the
"Additional requirements (from comments)" items.

The numbering is sequential, with no gaps, in the order the items
appear in the context file.

If during grooming a requirement contains several different things —
split it into sub-items: REQ-5a, REQ-5b. The main number is kept,
sub-items are added.

These IDs are permanent — they do not change across all the next skills
in the chain. REQ-5 in the grooming file is the same REQ-5 in the
checklist, in the code review, and onward.

## Grooming method

For each requirement from the context file go through four questions:

1. **Can I test this?**
   Is there enough specificity: expected result, screen, state,
   condition, trigger. If not — the requirement is uncovered.

2. **What do I need for it?**
   Is it clear which data, access, role, system state are needed. If it
   is not clear from the context — a clarifying question.

3. **Where can this break?**
   Check through the lens of test design:
   - Boundary values: is it described what happens at and beyond the
     boundary
   - States and transitions: are all states described, all valid and
     invalid transitions between them
   - Combinations of conditions: if there are several conditions (role
     + status + data state), is the behavior described for each
     combination
   - User classes: is behavior separated for different roles, account
     types, access levels
   - Negative scenarios: what if the data is empty, deleted, in an
     invalid state, the action is cancelled
   - Concurrent actions: what if the state changed while the user is on
     the page

4. **What is not specified?**
   - Alternative and exceptional scenarios (the main flow exists, but
     what if something went wrong?)
   - Dependencies between features: does the change affect other flows
   - Contradictions between requirements in the context file

Broad or vague wording is a signal to check deeper. Examples: "where
applicable", "all places", "update the wording", "appropriate
messages", "handle", "support".

### Application rules

- Each requirement goes through all 4 questions, but record a result
  only where the question is relevant. Do not generate problems
  artificially for simple or unambiguous requirements.
- After checking individual requirements — check related groups:
  requirements that describe one flow, one screen, or one feature. At
  the seams between related requirements there can be gaps not visible
  when checking each one alone.
- If several requirements describe variants of one behavior — check
  that all variants are covered, including those not stated explicitly
  (negative, alternative, exceptional).
- Record only what really affects the implementation, testing, or
  acceptance of the task. A question that will not change the code, the
  test, or the acceptance result — do not record.
- Do not generate questions about:
  - implementation details the developer decides (architecture,
    libraries, cache, performance) unless it is stated in the
    requirements as a requirement
  - standard browser or platform behavior (opening a tab, scroll,
    focus)
  - UX details that do not affect functionality (animations, timings)
    unless they are stated in the requirements
  - general infrastructure (server errors, timeouts, retries, logging,
    network failures) unless the ticket is specifically about this
  - things that are obvious from the task or product context — do not
    re-ask what is already clear

## Order of work

1. Read the context file in full, including the "Additional
   requirements (from comments)" and "⚠️ Conflicts to resolve" sections.
2. If the context file has a "⚠️ Conflicts to resolve" section, raise
   every conflict in it with the user first — each conflict is a
   Contradiction finding (Confluence acceptance criteria vs Jira
   Description). These must be resolved before the final requirements
   can be trusted.
3. If something is unclear about the product, feature, or task
   context — ask the user before starting grooming. Do not start
   grooming until you understand the context. Do not ask about what is
   obvious from the context or about what does not affect requirement
   grooming.
4. After getting the answers — start grooming the requirements.
5. Number all requirements: REQ-1, REQ-2...REQ-N (Requirements section
   first, then the Additional requirements from comments).
6. Groom each requirement through the grooming method.
7. In the chat, write only the problem requirements, grouped by
   category: questions, contradictions, potential bugs, uncovered
   requirements, risks. One REQ can appear in several categories. Each
   problem — briefly: the essence of the problem and what is needed to
   resolve it. Do not mention covered requirements.

   Chat format:

   ### Questions
   - REQ-3: <essence of the problem> → <what is needed to resolve it>

   ### Contradictions
   - REQ-5: <essence> → <what is needed>

   ### Potential bugs
   - REQ-7: <essence> → <what is needed>

   ### Uncovered requirements
   - REQ-9: <essence> → <what is needed>

   ### Risks
   - REQ-7: <essence> → <what is needed>

   Do not write empty categories.
8. Wait for the user's answers, align on the decisions. If the user
   cannot answer or says to skip — leave the requirement as is, without
   changes. For an unresolved conflict, keep BOTH versions in the
   requirement text and mark it "(unresolved conflict)" so it is not
   silently lost.
9. Produce the final list: covered requirements unchanged + problem
   requirements rewritten taking the alignment into account. Copy the
   "Goal" section from the context file unchanged — the next skills
   need it to understand the context.
10. Save one file — a clean numbered list of finished requirements.

## Output file

Create the file <ISSUEKEY>-requirements.md in the working directory and
give it to the user for download.

The file stays in the working directory — the next skill in the same
chat picks it up automatically.

If the file already exists — delete it completely and create a new one.

The template is in references/output-template.md.

## Final response

After saving the file, report the path to the saved file.
